# Repo Scan

## 概览

用这个 skill 处理反复出现的“研究这个仓库/插件/工具”类任务。目标是稳定回答：它是什么、给谁用、技术栈是什么、安装和依赖最少占多少空间、运行需要什么、风险在哪里、是否值得安装或二开，以及最小下一步是什么。

除必要技术名词外，默认全部用中文输出。

## 执行规则

- 第一轮默认只读。评估阶段不要 clone、安装、运行、购买、登录或修改本地文件。
- 优先使用一手证据：仓库 README、docs、release、issue、扩展商店页面、官方文档、包元数据和仓库源码。
- 对最新版本、更新时间、支持平台、价格、商店政策等可能变化的信息，可用时必须核验最新来源。
- 区分“已确认事实”和“推断”。证据不足的内容标为 `待确认`。
- 用户问“能不能安装”或“能不能二开”时，先回答前置条件和风险；只有用户明确确认后再实施。
- 涉及账号、抓取、自动化、代理、平台风控时，要写清合规风险和账号安全风险，不给绕过式结论。
- 优先给最小下一步：继续读文档、本地 smoke test、隔离目录试装、做小范围 pilot，或跳过。

## 工作流

1. 识别对象：
   - 名称、URL、作者/发布者、license。
   - 类型：仓库、扩展、CLI、skill、Web 应用、库、服务或混合项目。
   - 用户意图：了解、安装、接入、二开、对比或做类似工具。

2. 读取最小证据：
   - README 和 docs，用于判断用途与使用流程。
   - `package.json`、`pyproject.toml`、`requirements.txt`、Docker 文件、manifest 或扩展配置，用于判断技术栈、运行环境和安装条件。
   - release artifact、Docker image、模型下载说明、依赖锁文件和安装文档，用于估算项目本体、运行环境、依赖和模型的最小空间。
   - 最近 commit、release、issue，用于判断维护状态。
   - license 和政策敏感部分，尤其是安装、抓取、媒体下载、账号自动化或用户数据处理相关内容。

3. 说明它做什么：
   - 用一段话说明核心能力。
   - 说明主要工作流。
   - 说明输入、输出，以及数据保存或发送到哪里。
   - 说明它不做什么，尤其是项目名称暗示的能力超过代码实际支持范围时。

4. 梳理技术栈：
   - 语言与运行时：例如 Python、Node.js、Go、Rust、Browser Extension。
   - 前端/后端框架：例如 React、FastAPI、Next.js、Electron。
   - 核心依赖：模型、数据库、浏览器扩展 API、OCR、ASR、Docker、GPU、外部 SaaS。
   - 部署方式：本地运行、Docker、云服务、浏览器扩展、CLI、桌面应用。
   - 外部接口：API key、OAuth、cookie、账号、代理或第三方服务。

5. 估算空间占用：
   - `项目本体最小占用`：源码、release 包、浏览器扩展包、二进制文件或 Docker image 的最小体积。
   - `依赖最小占用`：语言运行时、包依赖、系统依赖、数据库、浏览器、Docker layer 等额外空间。
   - `模型/数据最小占用`：LLM、OCR、ASR、embedding、向量库、示例数据、缓存或权重文件。
   - `首次安装最低预留空间`：项目本体 + 依赖 + 模型/数据 + 安装缓存的保守估算。
   - 证据不足时不要编造精确数字，写 `待确认`，并说明需要通过 release 文件大小、Docker image inspect、包管理器 dry-run、模型下载页面或实际试装确认。

6. 评估实际使用价值：
   - 适合的用户和场景。
   - 本地要求：OS、语言/runtime、API key、模型 key、浏览器扩展、Docker、GPU、数据库或外部服务。
   - 安装难度：低 / 中 / 高。
   - 维护风险：低 / 中 / 高。
   - 隐私与安全风险：低 / 中 / 高。

7. 决定下一步：
   - `install`：只有前置条件清楚、风险可接受，并且用户明确要求安装时才进入安装。
   - `pilot`：有价值，但需要先隔离测试。
   - `adapt`：二开范围清楚，可以做小改造。
   - `clean-room rebuild`：直接复用有风险，但可以基于公开行为重新实现。
   - `skip`：项目废弃、信息不清、风险过高、重复度高，或不值得投入安装成本。

## 输出格式

多数报告使用以下结构：

1. `结论`
2. `它是做什么的`
3. `适合谁 / 适用场景`
4. `技术栈`
5. `最小空间占用`
6. `怎么用 / 运行条件`
7. `安装或接入难点`
8. `二开价值`
9. `风险`
10. `建议下一步`
11. `证据来源`

结论要直接。如果用户只问一个窄问题，先回答该问题，再补证据。

## 二开价值判断

用具体理由判断二开价值：

- `高`：项目活跃，license 允许或可走 clean-room 路线，领域稳定，架构可复用，并且用户有匹配工作流。
- `中`：概念有用，但安装、维护、政策风险或代码质量需要先做小范围 pilot。
- `低`：偏 demo、长期不维护、license 不清晰、依赖重、抓取脆弱、账号风险高，或已有 skill/tool 已充分覆盖。

如果建议 clean-room rebuild，只描述行为和架构，不复制专有代码、prompt、素材或私有实现细节。

## 安装前闸门

安装或 clone 前必须确认：

- 目标目录。
- 是否允许联网和下载。
- 磁盘剩余空间是否满足：项目本体、依赖、模型/数据、安装缓存的最低预留空间。
- 是否需要凭证、API key、浏览器 cookie 或账号。
- 是否会影响 Node、Python、Homebrew、Docker、浏览器扩展或 shell PATH 等全局工具。
- 安装失败时如何回滚。

如果用户明确说“只做检查”，停在证据和结论，不安装。
