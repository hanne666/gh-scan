# GitHub 仓库扫描

用于处理“研究这个仓库/插件/工具”类任务。目标是稳定回答：它是什么、给谁用、技术栈是什么、安装和依赖最少占多少空间、运行需要什么、风险在哪里、是否值得安装或二开，以及最小下一步是什么。

除必要技术名词外，默认全部用中文输出。

## 必读参考

如果需要完整评估标准，读取本 skill 的参考文件：

- `references/criteria.md`

如果是在套件真源目录中维护，也可以对照 `skills/gh-scan/shared/repo-scan-criteria.md`。

## 执行规则

- 第一轮默认只读。评估阶段不要 clone、安装、运行、购买、登录或修改本地文件。
- 优先使用一手证据：仓库 README、docs、release、issue、扩展商店页面、官方文档、包元数据和源码。
- 对最新版本、更新时间、支持平台、价格、商店政策等可能变化的信息，必须核验最新来源。
- 区分“已确认事实”和“推断”。证据不足的内容标为 `待确认`。
- 用户问“能不能安装”或“能不能二开”时，先回答前置条件和风险；只有用户明确确认后再实施。
- 涉及账号、抓取、自动化、代理、平台风控时，写清合规风险和账号安全风险。
- 优先给最小下一步：继续读文档、本地 smoke test、隔离目录试装、做小范围 pilot，或跳过。

## 工作流

1. 识别对象：
   - 名称、URL、作者/发布者、license。
   - 类型：仓库、扩展、CLI、skill、Web 应用、库、服务或混合项目。
   - 用户意图：了解、安装、接入、二开、对比或做类似工具。

2. 读取最小证据：
   - README 和 docs。
   - `package.json`、`pyproject.toml`、`requirements.txt`、Docker 文件、manifest 或扩展配置。
   - release artifact、Docker image、模型下载说明、依赖锁文件和安装文档。
   - 最近 commit、release、issue。
   - license 和政策敏感部分。

3. 说明它做什么：
   - 一段话说明核心能力。
   - 说明主要工作流。
   - 说明输入、输出，以及数据保存或发送到哪里。
   - 说明它不做什么。

4. 梳理技术栈：
   - 语言与 runtime。
   - 前端/后端框架。
   - 核心依赖。
   - 部署方式。
   - 外部接口：API key、OAuth、cookie、账号、代理或第三方服务。

5. 估算空间占用：
   - 项目本体最小占用。
   - 依赖最小占用。
   - 模型/数据最小占用。
   - 首次安装最低预留空间。
   - 证据不足时写 `待确认`，不要编造精确数字。

6. 评估实际使用价值：
   - 适合的用户和场景。
   - 本地要求。
   - 安装难度：低 / 中 / 高。
   - 维护风险：低 / 中 / 高。
   - 隐私与安全风险：低 / 中 / 高。

7. 决定下一步：
   - `install`
   - `pilot`
   - `adapt`
   - `clean-room rebuild`
   - `skip`

## 输出格式

多数报告使用：

1. `结论`
2. `它是做什么的`
3. `适合谁 / 适用场景`
4. `技术栈`
5. `最小空间占用`
6. `怎么用 / 运行条件`
7. `安装或接入难点`
8. `二开价值`
9. `风险`
10. `建议下一步`
11. `证据来源`

如果用户只问一个窄问题，先回答该问题，再补证据。
